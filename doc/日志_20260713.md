## 附录：SLAM 建图方案对比

### 方案 A：自动 move_dir + 避障
自动发送方向指令让小车移动固定距离，SLAM判定到达。

| 优点 | 缺点 |
|------|------|
| 全自动，不需要人操作 | 编码器系数不准（encoder_to_meter=0.001偏大），距离控制差 |
| 避障内置 | SLAM位置更新慢（2-5Hz），容易过冲 |
| | 旋转检测IMU yaw不稳定 |
| | 原厂app.py总抢串口 |

**结论：❌ 不可靠，放弃**

---

### 方案 B：键盘 WASD 直控
小车运行 keyboard_control.py，通过 WASD 按键遥控。

| 优点 | 缺点 |
|------|------|
| 自己控制运动，想停就停 | termios 只能在 NoMachine 终端用 |
| | 松手检测不够灵敏 |
| | 串口被 app.py 抢占后需手动恢复 |
| | 依赖 keyboard 库（Linux需root，改用termios） |

**结论：⚠️ 可用但不稳定**

---

### 方案 C：App遥控 + position_recorder（推荐 ✅）
用户通过原厂 app 或鸿蒙App遥控小车，position_recorder.py 只读 SLAM 位置不碰串口。

| 优点 | 缺点 |
|------|------|
| ✅ **各干各的，完全不冲突** | 需要两个人配合操作 |
| ✅ 不需要串口，不会被 app.py 干扰 | |
| ✅ position_recorder 只读 ROS2 TF，位置准确 | |
| ✅ 用户用车本身的遥控方式，手感好 | |

**结论：✅ 推荐方案**
